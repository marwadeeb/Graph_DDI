RM1 — Explainability
How does the system explain its predictions? Partial
Every prediction comes with a source label and a human-readable explanation.
1

Dict lookup — 100 % interpretable ✓ Done

If a pair is found in DrugBank, the response includes the exact DrugBank sentence describing the mechanism: e.g. "The risk or severity of bleeding can be increased when Warfarin is combined with Aspirin." This is direct database retrieval — no model inference, fully auditable.

2

Logistic Regression — feature attribution ✓ Done

LR coefficients (shown on the Results page) reveal which physicochemical properties drive predictions: CYP450 enzyme overlap, ATC category co-membership, molecular weight difference. Any prediction can be traced to specific features.

3

GNN — post-hoc explainability Planned

GNN predictions are flagged with a probability and a disclaimer ("not documented in DrugBank — model prediction only"). GNNExplainer (attention-weighted subgraph attribution) is planned after model delivery. Until then, each GNN prediction includes the confidence score and source label.

4

Source transparency — always shown ✓ Done

Every API response includes a source field: documented (DrugBank hit), gnn_predicted (model inference), or not_found (no evidence). Users always know whether they are looking at established data or a model prediction.

What we do about it:

📊 Document and disclose

The coverage gap is measured and reported here explicitly. Users are informed that predictions for less-studied drug categories (VARIOUS, ANTIPARASITIC) should be treated with lower confidence.

🧪 Per-category error analysis

GNN AUC is evaluated per ATC category — performance gaps are visible above, not hidden inside a single aggregate metric. Underperforming categories align directly with the training-data sparsity documented in the bias table.

⚠️ Prediction disclaimers

GNN predictions (not in DrugBank) are always flagged with a disclaimer recommending clinical verification. The system never presents model output as confirmed medical fact.

🔗 Dict lookup first

The primary path is direct DrugBank lookup — a model is only invoked when no documented interaction exists. This means the most safety-critical known interactions are always handled by ground truth, not inference.

RM3 — Privacy
Why is this system privacy-preserving by design? Done
The DDI Checker handles no patient data at any stage. Privacy is guaranteed structurally, not just by policy.
  • ✅
    Public data only. All training data is DrugBank Full Database (CC BY-NC 4.0) — a curated, publicly available pharmacological knowledge base. No electronic health records, prescriptions, or patient identifiers are used at any stage.
  • ✅
    No user data stored. Drug pair queries submitted via the web interface or API are not logged, stored, or used for retraining. Each request is ephemeral.
  • ✅
    No re-identification risk. Drug names are not linked to any individual. Querying "Warfarin + Aspirin" reveals nothing about who is asking — it is a substance-level query, not a patient-level query.
  • ✅
    Open-source deployment. The full pipeline is reproducible from publicly available data. No proprietary patient cohorts, no data-sharing agreements required. DrugBank requires registration at drugbank.ca — the raw XML is not redistributed.
  • ✅
    GDPR / clinical safety posture. The system is explicitly framed as a decision-support tool, not a clinical tool. Predictions include mandatory disclaimers recommending professional consultation. No personal data is processed, so GDPR subject-access rights do not apply.

Design decisions behind the failures — and why they are correct:

🚫 Brand names rejected by design

"Tylenol" → not found. Brand names vary by country, change over time, and can map to multiple generics. Accepting an ambiguous brand name risks silent misidentification in a clinical safety context — returning wrong interaction data is more dangerous than returning not found.

🚫 Misspellings rejected by design

"wafrarin" → not found. Fuzzy matching would risk accepting a wrong drug name in a high-stakes safety context. The system returns not found and asks users to verify spelling — safer than guessing.

✅ Case-insensitive matching

"WARFARIN", "warfarin", "Warfarin" all resolve correctly. Drug names have no meaningful capitalisation — rejecting uppercase would be a usability failure with no safety benefit.

✅ Common synonyms supported

"aspirin" → Acetylsalicylic acid, "adrenaline" → Epinephrine. The synonym table covers the most clinically common alternative names from the DrugBank drug_attributes.csv file.

✅ DrugBank IDs accepted

DB00682 → Warfarin. API callers (developers, clinical systems) often work with structured identifiers rather than free-text names.

✅ Empty / nonsense inputs raise errors

Empty strings and non-drug text return drug_not_found rather than a silent failure. The system surfaces the problem to the caller rather than returning a spurious result.

Improvement path: Adding a commercial synonym table (Tylenol, Advil, Lasix, etc.) and a fuzzy-match fallback with a confirmation step would increase the pass rate from 60% to ~90% without compromising safety — the confirmation step prevents silent misidentification.
